Analyze the code through the following lenses:

1. **System Integrity & Layering**:
   - Evaluate if the changes respect established architectural boundaries.
   - Detect "leaky abstractions" where implementation details bleed into higher layers.
   - Ensure a clean separation between domain logic, infrastructure, and presentation.

2. **SOLID & Design Patterns Enforcement**:
   - Identify violations of SOLID principles, particularly SRP (Single Responsibility) and DIP (Dependency Inversion).
   - Evaluate the appropriateness of used design patterns (e.g., Factory, Strategy, Observer).

3. **Modularity, Coupling & Cohesion**:
   - Assess the "blast radius" of the changes. Does a small change require ripple effects elsewhere?
   - Look for tight coupling between modules that should be independent.
   - Evaluate if the new components have high cohesion.

4. **Domain-Driven Design (DDD) Rigor**:
   - Verify if domain concepts are accurately captured in Entities and Value Objects.
   - Check if Aggregate boundaries are respected and if Repositories are used correctly (hiding persistence logic).

5. **Error Handling Architecture**:
   - Ensure error handling is not just "catching exceptions" but follows a strategic pattern (e.g., Result objects, centralized middleware, or standardized error codes).

6. **Future-Proofing & Tech Debt**:
   - Identify shortcut decisions that might lead to significant technical debt.
   - Evaluate if the code is "over-engineered" for the current requirements.
